| Vercel | - | Next.js 最佳部署平台，免費方案足夠 |

### 2.2 前端輔助套件

| 套件 | 用途 |
|------|------|

| react-medium-image-zoom | 圖片點擊放大 |
| nodemailer | 寄送備份 Email |

### 2.3 架構設計原則

**App Router（而非 Pages Router）：**
Next.js 13 之後的新架構。每個資料夾代表一個路由，每個 `page.tsx` 是頁面元件。
優點：支援 Server Components（減少客戶端 JS）、Layouts 巢狀、更好的 SEO。

**Context 管理狀態：**
- `AuthContext` — 全域登入狀態
- `ThemeContext` — 深色/淺色主題
- `I18nContext` — 多語系切換

**CSS Modules：**
每個元件有對應的 `.module.css`，避免樣式污染。

### 2.4 環境變數架構

```
.env.local（本機開發，不進 git）
├── NEXT_PUBLIC_SUPABASE_URL      # 前後端共用
├── NEXT_PUBLIC_SUPABASE_ANON_KEY # 前後端共用（公開）
├── SUPABASE_SERVICE_ROLE_KEY      # 僅後端（備份 API 用）
├── GMAIL_USER                     # 備份寄件帳號
├── GMAIL_PASS                     # Gmail 應用程式密碼
```

`NEXT_PUBLIC_` 前綴：代表這個變數可以在瀏覽器端讀取，**不可存放機密資訊**。
沒有前綴的變數：只能在 Server Side 讀取，安全。

### 3.2 Row Level Security（RLS）設計

**RLS 是什麼：**
PostgreSQL 的安全機制，讓每一行資料都有「誰能讀、誰能寫」的控制。
Supabase 預設 RLS 關閉，但開啟後若沒有設定 Policy，任何人都無法存取。

**三層權限架構：**
```
未登入訪客  → 讀取已發布的公開資料（status = 'published'）
登入會員    → 讀取/修改自己的個人資料
管理員      → 完整 CRUD 所有資料
```

**管理員識別方式：**
使用 JWT 中的 email 欄位做比對：
```sql
auth.jwt()->>'email' = '您的管理員Email'
```

**重要陷阱：**
- `FOR ALL USING (...)` 只控制「讀取條件」
- Insert 和 Update 需要額外的 `WITH CHECK (...)` 才能生效
- 工作坊管理員 Policy 沒有 `WITH CHECK` 會導致新增失敗（這是第七次升級的 bug）

### 3.3 Prompts 審核狀態流程

```
前台提交 → pending_review
管理員審核通過 → published
管理員拒絕 → rejected
管理員存草稿 → draft
```

**初始設計的 Bug：**
舊版資料表對 status 有 CHECK 約束，只允許特定值。加入 `pending_review` 時
需要先 DROP CONSTRAINT 再重新 ADD CONSTRAINT。
同時要把之前誤存為 `'pending'`（非 `'pending_review'`）的資料修正：
```sql
UPDATE prompts SET status = 'pending_review' WHERE status = 'pending';
```

### 5.1 前台展示頁面

```typescript
    </div>
  )
```

## 6. 會員系統實作

### 6.1 架構設計

```
auth.users（Supabase 內建）
    ↓ 新使用者觸發器
profiles（自訂 - 儲存額外個人資料）
    ↓ 同步
points_log（積分變動記錄）
```

### 6.2 Google OAuth 登入流程

```
1. 使用者點擊「Google 登入」
2. 導向 Google OAuth 同意頁
3. 使用者授權後，Google 導回 Supabase callback
4. Supabase 建立 auth.users 記錄
5. 觸發器 on_auth_user_created 自動執行
6. 在 profiles 建立會員資料 + 在 points_log 記錄 200 積分
7. 前台顯示登入成功狀態
```

### 6.3 觸發器（最終版本）

```sql
CREATE OR REPLACE FUNCTION public.handle_new_user()
RETURNS trigger AS $$
BEGIN
  INSERT INTO public.profiles (id, email, username, points, level)
  VALUES (
    NEW.id,
    NEW.email,
    COALESCE(
      NEW.raw_user_meta_data->>'full_name',  -- Google 提供的全名
      NEW.raw_user_meta_data->>'name',       -- 備選
      split_part(NEW.email, '@', 1)          -- 最後備選：email @ 前的部分
    ),
    200,      -- 新會員贈送積分
    'normal'  -- 預設等級
  )
  ON CONFLICT (id) DO NOTHING;  -- 避免重複插入（重要！）

  INSERT INTO public.points_log (user_id, action, points)
  VALUES (NEW.id, '新會員註冊', 200);
  RETURN NEW;
END;
$$ LANGUAGE plpgsql SECURITY DEFINER;
```

**`ON CONFLICT (id) DO NOTHING` 的重要性：**
如果觸發器因為任何原因被呼叫兩次（例如 OAuth 重試），沒有這行會因為主鍵衝突而報錯。

**`SECURITY DEFINER` 的作用：**
讓這個函數以「資料庫擁有者」的權限執行，而非呼叫者的權限，才能繞過 RLS 寫入 profiles。

### 6.4 AuthContext 設計

```typescript
// src/contexts/AuthContext.tsx
const AuthContext = createContext<{
  session: Session | null
  user: User | null
  loading: boolean
  signOut: () => Promise<void>
}>({...})

export function AuthProvider({ children }) {
  const [session, setSession] = useState<Session | null>(null)
  const [user, setUser] = useState<User | null>(null)
  const [loading, setLoading] = useState(true)

  useEffect(() => {
    // 1. 取得初始 session
    supabase.auth.getSession().then(({ data }) => {
      setSession(data.session)
      setUser(data.session?.user ?? null)
      setLoading(false)
    })

    // 2. 訂閱 auth 狀態變化
    const { data: { subscription } } = supabase.auth.onAuthStateChange(
      (event, session) => {
        setSession(session)
        setUser(session?.user ?? null)
      }
    )

    return () => subscription.unsubscribe()
  }, [])
  // ...
}
```

### 6.5 會員等級設計

**最終版本（三種類型）：**

| 等級 | 英文值 | 說明 |
|------|--------|------|
| 一般會員 | `normal` | 預設，所有新會員 |
| 贊助會員 | `sponsor` | 手動由管理員設定 |
| 訂閱會員 | `subscriber` | 手動由管理員設定 |

**設計決策：**
早期考慮過積分升級制（銅/銀/金/鑽石），但後來改為手動管理的三類型，
原因是積分不應該與等級掛鉤，等級應該反映實際付費或贊助狀態。

## 7. 後台管理系統

### 7.1 路由保護（Middleware）

```typescript
// src/middleware.ts
export async function middleware(request: NextRequest) {
  const res = NextResponse.next()
  const supabase = createServerClient(...)

  const { data: { session } } = await supabase.auth.getSession()

  const isAdmin = request.nextUrl.pathname.startsWith('/admin')
  const isLoginPage = request.nextUrl.pathname === '/admin/login'

  // 未登入 + 存取後台 → 導向登入頁
  if (isAdmin && !isLoginPage && !session) {
    return NextResponse.redirect(new URL('/admin/login', request.url))
  }

  // 已登入 + 存取登入頁 → 導向後台首頁
  if (isLoginPage && session) {
    return NextResponse.redirect(new URL('/admin/dashboard', request.url))
  }

  return res
}

export const config = {
  matcher: ['/admin/:path*']
}
```

### 7.2 AdminShell 設計

AdminShell 除了提供 Layout，還負責：
1. 再次驗證 session（Double-check 安全）
2. 設定 5 秒 timeout：若 5 秒內無法確認身分，自動導向登入頁

```typescript
// 防止畫面閃白：在確認 session 之前顯示 loading
```

### 7.3 後台 CRUD 通用模式

每個後台管理頁面都遵循相同模式：

```typescript
// 狀態
const [items, setItems] = useState([])
const [editing, setEditing] = useState(null)  // null = 新增, object = 編輯
const [form, setForm] = useState(defaultForm)

// 讀取
async function fetchItems() {
  const { data } = await supabase.from('table').select('*').order('created_at')
  setItems(data || [])
}

// 新增 / 更新（同一個函數）
async function handleSave() {
  if (editing) {
    await supabase.from('table').update(form).eq('id', editing.id)
  } else {
    await supabase.from('table').insert(form)
  }
  fetchItems()
  setEditing(null)
  setForm(defaultForm)
}

// 刪除
async function handleDelete(id) {
  if (!confirm('確定刪除？')) return
  await supabase.from('table').delete().eq('id', id)
  fetchItems()
}
```

## 8. 自動化備份系統

### 8.1 架構

```
每週日 08:00（台灣時間）
    ↓
cron-job.org 發出 GET 請求
    ↓
/api/backup/cron（驗證 Bearer Token）
    ↓
Supabase Service Role Key 讀取所有資料表
    ↓
整理成 JSON
    ↓
nodemailer（Gmail SMTP）發送附件
    ↓
管理員信箱收到備份
```

### 8.2 API 路由設計

```typescript
// src/app/api/backup/cron/route.ts
export async function GET(request: Request) {
  // 1. 驗證 Bearer Token
  const authHeader = request.headers.get('Authorization')
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return new Response('Unauthorized', { status: 401 })
  }

  // 2. 用 Service Role Key 讀取所有資料表（bypass RLS）
  const supabase = createClient(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.SUPABASE_SERVICE_ROLE_KEY!  // 必須是 service role，不是 anon key
  )

  const tables = ['projects', 'prompts', 'skills', 'workshops', ...]
  const backup = {}
  for (const table of tables) {
    const { data } = await supabase.from(table).select('*')
    backup[table] = data
  }

  // 3. 發送 Email
  const transporter = nodemailer.createTransport({
    service: 'gmail',
    auth: {
      user: process.env.GMAIL_USER,
      pass: process.env.GMAIL_PASS  // 應用程式密碼，非登入密碼
    }
  })

  await transporter.sendMail({
    from: process.env.GMAIL_USER,
    to: process.env.BACKUP_EMAIL_TO,
    subject: `資料庫備份 ${new Date().toLocaleDateString('zh-TW')}`,
    attachments: [{
      filename: 'backup.json',
      content: JSON.stringify(backup, null, 2)
    }]
  })

  return new Response('OK')
}
```

### 8.3 安全設計

- `CRON_SECRET`：防止陌生人亂呼叫備份 API
- 備份 API 只有 `GET` 方法，不修改任何資料
- `SUPABASE_SERVICE_ROLE_KEY` 只用在 Server Side，不暴露給瀏覽器
- Gmail 應用程式密碼：即使被竊取，只能寄信，無法登入 Google 帳號

## 11. 除錯記錄（常見錯誤與解法）

### 11.1 RLS 相關錯誤

**症狀：** 管理員在後台新增工作坊時出現錯誤，但讀取正常
**根本原因：** `FOR ALL USING (...)` 只控制 SELECT/DELETE 的過濾條件
**解法：** 需要額外加 `WITH CHECK (...)` 才能控制 INSERT/UPDATE
```sql
-- 錯誤的寫法（工作坊無法新增）
CREATE POLICY "Admin full workshops" ON workshops
  FOR ALL USING (auth.jwt()->>'email' = '您的管理員Email');

-- 正確的寫法
CREATE POLICY "Admin full workshops" ON workshops
  FOR ALL
  USING (auth.jwt()->>'email' = '您的管理員Email')
  WITH CHECK (auth.jwt()->>'email' = '您的管理員Email');
```

---

**症狀：** Google 登入後前台建立 Profile 失敗
**根本原因：** 缺少 `Users insert own profile` Policy
**解法：**
```sql
CREATE POLICY "Users insert own profile" ON profiles
  FOR INSERT WITH CHECK (auth.uid() = id);
```

---

### 11.2 CHECK 約束錯誤

**症狀：** 提交提示詞時出現 `violates check constraint prompts_category_check`
**根本原因：** 舊版 prompts 資料表對 `category` 欄位有 CHECK 限制，只允許英文固定值
**解法：** 移除限制，允許任意分類名稱
```sql
ALTER TABLE prompts DROP CONSTRAINT IF EXISTS prompts_category_check;
```

---

**症狀：** 前台提交提示詞出現 `violates check constraint prompts_status_check`
**根本原因：** `status` 的 CHECK 限制沒有包含 `pending_review`
**解法：**
```sql
ALTER TABLE prompts DROP CONSTRAINT IF EXISTS prompts_status_check;
ALTER TABLE prompts ADD CONSTRAINT prompts_status_check
  CHECK (status IN ('pending_review', 'published', 'draft', 'rejected'));
```

---

### 11.3 Auth 相關錯誤

**症狀：** `redirect_uri_mismatch`
**根本原因：** Google GCP 設定的重新導向 URI 與 Supabase callback URL 不一致
**解法：** 確認 GCP 中的 URI 格式：`https://[PROJECT_ID].supabase.co/auth/v1/callback`

---

**症狀：** 本機無法用 Google 登入
**根本原因：** GCP 未加入 `http://localhost:3000` 為允許的 JavaScript 來源
**解法：** 到 GCP → 憑證 → OAuth 用戶端 → 已授權的 JavaScript 來源，加入 `http://localhost:3000`

---

### 11.4 備份相關錯誤

**症狀：** `Missing credentials for "PLAIN"`
**根本原因：** Vercel 環境變數 `GMAIL_USER` 或 `GMAIL_PASS` 未填入，或填入後未重新部署
**解法：** 確認 Vercel 環境變數都已填入，然後 Redeploy

---

**症狀：** Cron Job 呼叫備份 API 收到 401 Unauthorized
**根本原因：** cron-job.org 設定的 Authorization Header 格式錯誤
**正確格式：** `Bearer 你的CRON_SECRET`（注意 Bearer 後面有一個空格）

---

### 11.5 Navbar 與 AdminShell 衝突

**症狀：** 後台頁面 Navbar 出現閃動或雙重渲染
**根本原因：** 前台的 `AppShell`（包含 Navbar）包住了後台 Layout
**解法：** 後台 `/admin/layout.tsx` 要直接使用 `AdminShell`，不經過 `AppShell`

---

### 11.6 觸發器衝突

**症狀：** Google 登入時提示 `duplicate key value violates unique constraint "profiles_pkey"`
**根本原因：** 觸發器被呼叫兩次，第二次嘗試插入相同 id
**解法：** 在 INSERT 中加入 `ON CONFLICT (id) DO NOTHING`

## 13. 重要觀念整理

### 13.1 Supabase Client vs Server

| | `supabase.ts`（Client） | `supabaseServer.ts`（Server） |
|---|---|---|
| 執行位置 | 瀏覽器 | Next.js Server |
| 憑證 | anon key | service_role key（可 bypass RLS） |
| Session | 從 localStorage/Cookie | 從 Cookie |
| 使用場景 | 前台頁面、Client Component | API Routes、Server Component |

### 13.2 `'use client'` vs Server Component

Next.js App Router 預設所有元件是 **Server Component**（伺服器端渲染）。
加上 `'use client'` 才變成客戶端元件，才能使用：
- `useState`, `useEffect` 等 React Hooks
- 瀏覽器 API（`window`, `localStorage`）
- 事件監聽（`onClick` 等）

### 13.3 RLS Policy 語法對照

```sql
-- 讀取控制（WHERE 條件）
CREATE POLICY "..." ON table FOR SELECT USING (條件);

-- 寫入控制（INSERT 的值必須符合）
CREATE POLICY "..." ON table FOR INSERT WITH CHECK (條件);

-- 同時控制讀取和寫入
CREATE POLICY "..." ON table FOR ALL
  USING (讀取條件)
  WITH CHECK (寫入條件);
```

### 13.4 環境變數三層架構

```
開發環境            → .env.local（本機，不進 git）
正式環境（Vercel）  → Vercel Dashboard > Settings > Environment Variables
CI/CD（自動）       → 每次 Push，Vercel 使用已設定的環境變數自動部署
```

### 13.5 Cron Job 時區

- Vercel Cron 和 cron-job.org 都使用 **UTC 時間**
- 台灣是 UTC+8，所以：
  - UTC 00:00 = 台灣 08:00
  - `0 0 * * 0`（UTC週日凌晨）= 台灣週日早上 8 點

```bash
# 日常開發流程
git add .
git commit -m "功能說明"
git push

# Push 後 Vercel 自動部署（約 1~2 分鐘）
```
